# Спецификация проекта: Russian Text-to-SQL Assistant

## 1. Определение задачи (Task Definition)

Настоящий проект предназначен для создания и тонкой настройки (fine-tuning) компактной языковой модели, способной преобразовывать бизнес-вопросы на русском языке и схему реляционной базы данных (в диалекте SQLite) в валидные, безопасные SQL-запросы с пояснением на русском языке.

Модель получает на вход структурированное описание базы данных и текстовый вопрос пользователя, после чего генерирует ответ строго в формате JSON, содержащий сформированный SQL-запрос, текстовое объяснение логики выборки, список допущений (при наличии неоднозначности в вопросе) и уровень уверенности (confidence score).

---

## 2. Формат входных данных (Input Format)

Входные данные передаются модели в структурированном виде (JSON или строго размеченный системный промпт) со следующими обязательными и опциональными полями:

| Поле | Тип | Обязательность | Описание |
| :--- | :--- | :--- | :--- |
| `dialect` | `string` | Обязательно | Диалект SQL. На этапе MVP значение всегда `'sqlite'`. |
| `schema` | `string` | Обязательно | DDL-описание схемы базы данных (`CREATE TABLE` операторы) с указанием типов колонок, первичных и внешних ключей (`PRIMARY KEY`, `FOREIGN KEY`). |
| `business_question` | `string` | Обязательно | Бизнес-вопрос или запрос пользователя на русском языке. |
| `context` | `string` | Опционально | Дополнительный контекст, бизнес-ограничения, синонимы терминов или бизнес-правила (например, определение "активного клиента"). |

### Пример входного промпта
```json
{
  "dialect": "sqlite",
  "schema": "CREATE TABLE users (\n    id INTEGER PRIMARY KEY,\n    name TEXT NOT NULL,\n    email TEXT UNIQUE,\n    age INTEGER\n);\n\nCREATE TABLE orders (\n    id INTEGER PRIMARY KEY,\n    user_id INTEGER,\n    amount REAL NOT NULL,\n    created_at TEXT NOT NULL,\n    FOREIGN KEY (user_id) REFERENCES users(id)\n);",
  "business_question": "Покажи имена пользователей старше 30 лет, которые сделали заказы на сумму более 1000 рублей",
  "context": null
}
```

---

## 3. Формат выходных данных (Output Format)

Модель должна генерировать ответ строго в валидном JSON-формате без дополнительного текста вокруг (Markdown JSON block допускается при парсинге, но предпочтителен чистый JSON).

### JSON Schema
```json
{
  "sql": "SELECT ...",
  "explanation_ru": "Краткое объяснение логики запроса на русском языке",
  "assumptions": [
    "список принятых допущений при неоднозначности или отсутствии явных бизнес-правил"
  ],
  "confidence": "high|medium|low"
}
```

### Описание полей вывода
- `sql` (`string`): Строка с SQL-запросом. Для разрешенных запросов начинается с `SELECT`. Если запрос отклонен (отсутствует в scope или опасен), поле содержит пустую строку `""`.
- `explanation_ru` (`string`): Понятное краткое объяснение того, как работает запрос, какие таблицы объединяются и по каким условиям выполняется фильтрация.
- `assumptions` (`array of strings`): Список допущений. Если вопрос однозначный, передается пустой массив `[]`. Если вопрос содержит неопределенность (например, "активный пользователь"), модель явно перечисляют сделанные гипотезы.
- `confidence` (`enum`): Оценка уверенности модели. Допустимые значения: `"high"`, `"medium"`, `"low"`.

## 4. Область применения MVP (Phase 1 Scope)

### 4.1 Поддерживаемые возможности SQL (In Scope)
На первом этапе (MVP) модель должна уверенно поддерживать следующие конструкции SQLite dialect:
- **Оператор выборки:** Только `SELECT` запросы.
- **Соединение таблиц:** `JOIN` (`INNER JOIN`, `LEFT JOIN`, `RIGHT JOIN`).
- **Фильтрация:** Секция `WHERE` с операторами сравнения (`=`, `!=`, `<`, `>`, `<=`, `>=`), `LIKE`, `IN`, `BETWEEN`, `IS NULL`, `IS NOT NULL`, логическими операторами `AND`, `OR`, `NOT`.
- **Группировка и фильтрация групп:** `GROUP BY`, `HAVING`.
- **Сортировка и ограничение выборки:** `ORDER BY` (`ASC`, `DESC`), `LIMIT`.
- **Агрегатные функции:** `COUNT()`, `SUM()`, `AVG()`, `MIN()`, `MAX()`.
- **Работа с датой и временем:** Функции SQLite `DATE()`, `strftime()`.
- **Вложенные запросы:** Подзапросы в секциях `WHERE` (например, `IN (SELECT ...)`) и `FROM`.
- **Условная логика:** Выражения `CASE WHEN ... THEN ... ELSE ... END`.
- **Уникальность и псевдонимы:** `DISTINCT`, `AS` (aliases для таблиц и столбцов).

### 4.2 Явно исключено из MVP (Explicitly Out of Scope)
Следующие категории запросов **категорически запрещены** или отложены на будущие фазы:
- **Модификация данных и схемы (DML / DDL):** `INSERT`, `UPDATE`, `DELETE`, `DROP`, `CREATE`, `ALTER`, `TRUNCATE`. При попытке пользователя сгенерировать такие запросы модель должна возвращать `sql: ""` с объяснением отказа.
- **Объекты базы данных:** Создание и вызов хранимых процедур, триггеров, представлений (`CREATE VIEW`).
- **Оконные функции (Window Functions):** `OVER (PARTITION BY ... ORDER BY ...)` — планируется в Phase 2.
- **Операции над множествами:** `UNION`, `UNION ALL`, `INTERSECT`, `EXCEPT` — планируется в Phase 2.
- **Небезопасные / Деструктивные SQL-инструкции:** Вызов системных PRAGMA, экспорт/импорт файлов, мульти-запросы через `;`.

## 5. Критерии успеха (Success Criteria)

1. **Точность выполнения (Execution Accuracy, EX):**
   - Достижение $\ge 60\%$ точности выполнения SQL-запросов на независимом тестовом датасете (Held-out test set с новыми, не встречавшимися при обучении схемами БД).
   - Точность проверяется сравнением результатов выполнения сгенерированного SQL и эталонного SQL на реальной тестовой БД.
2. **Безопасность (Safety Compliance):**
   - 100% сгенерированных валидных SQL-запросов должны относиться только к категории `SELECT`.
   - Запросы `DELETE`, `UPDATE`, `DROP` и т.д. должны на 100% блокироваться моделью.
3. **Валидность синтаксиса и формата (JSON & SQL Syntax Validity):**
   - 100% ответов модели должны успешно парситься стандартным JSON-парсёром.
   - $\ge 90\%$ сгенерированных SQL-запросов должны компилироваться SQLite без синтаксических ошибок.
4. **Качество объяснений (Explanation Quality):**
   - Все текстовые пояснения в `explanation_ru` должны быть написаны на грамотном русском языке и соответствовать логике сгенерированного SQL-кода.

---

## 6. Технические ограничения и требования к инфраструктуре (Constraints)

- **Базовая модель (Base Model):** Модели класса ~3-4B параметров (например, Qwen3-4B / Llama-3.2-3B-Instruct или их русскоязычные адаптации, лицензия Apache-2.0).
- **Метод обучения:** Fine-tuning с использованием QLoRA (4-bit quantization, LoRA adapters).
- **Аппаратные ограничения (Hardware Target):** 
  - GPU: 1x NVIDIA GeForce RTX 5070 Ti (16 GB VRAM).
  - VRAM Budget при обучении: $\le 14\text{ GB}$ VRAM.
- **Инференс (Inference Performance):** 
  - Запуск на одиночном GPU.
  - Задержка генерации ответа (Latency): $< 5\text{ секунд}$ на один запрос.
- **Лицензирование и данные:**
  - Отсутствие проприетарных данных в обучающей выборке. Использование открытых датасетов (например, перевод/адаптация Spider, BIRD, WikiSQL) и синтетических данных, сгенерированных разрешенными Open-Weights моделями.
  - Обученный адаптер (LoRA weights) должен распространяться под пермиссивной лицензией (Apache 2.0 / MIT).

## 7. Примеры работы (Examples)

### 7.1 Корректный запрос (Simple Query)

#### Входные данные:
```json
{
  "dialect": "sqlite",
  "schema": "CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT NOT NULL, email TEXT UNIQUE, age INTEGER);\nCREATE TABLE orders (id INTEGER PRIMARY KEY, user_id INTEGER, amount REAL NOT NULL, created_at TEXT NOT NULL, FOREIGN KEY (user_id) REFERENCES users(id));",
  "business_question": "Покажи имена пользователей старше 30 лет, которые сделали заказы на сумму более 1000 рублей"
}
```

#### Ожидаемый вывод модели:
```json
{
  "sql": "SELECT DISTINCT t1.name FROM users AS t1 INNER JOIN orders AS t2 ON t1.id = t2.user_id WHERE t1.age > 30 AND t2.amount > 1000",
  "explanation_ru": "Запрос объединяет таблицы пользователей (users) и заказов (orders) по идентификатору пользователя. Фильтрация отбирает пользователей в возрасте строго больше 30 лет, у которых сумма отдельного заказа превышает 1000 рублей. Ключевое слово DISTINCT исключает дублирование имен при наличии нескольких подходящих заказов.",
  "assumptions": [],
  "confidence": "high"
}
```

---

### 7.2 Неоднозначный запрос (Ambiguous Query)

#### Входные данные:
```json
{
  "dialect": "sqlite",
  "schema": "CREATE TABLE clients (client_id INTEGER PRIMARY KEY, name TEXT, registration_date TEXT);\nCREATE TABLE transactions (transaction_id INTEGER PRIMARY KEY, client_id INTEGER, amount REAL, transaction_date TEXT, FOREIGN KEY (client_id) REFERENCES clients(client_id));",
  "business_question": "Найди самых активных клиентов за последний месяц"
}
```

#### Ожидаемый вывод модели:
```json
{
  "sql": "SELECT t1.client_id, t1.name, COUNT(t2.transaction_id) AS transaction_count, SUM(t2.amount) AS total_amount FROM clients AS t1 INNER JOIN transactions AS t2 ON t1.client_id = t2.client_id WHERE t2.transaction_date >= DATE('now', '-1 month') GROUP BY t1.client_id, t1.name ORDER BY transaction_count DESC LIMIT 10",
  "explanation_ru": "Поскольку понятие 'самый активный' не определено явно, активность рассчитана по количеству транзакций за последние 30 дней. Запрос выводит 10 клиентов с наибольшим числом транзакций за этот период, а также общую сумму их транзакций.",
  "assumptions": [
    "Под 'последним месяцем' понимается период в 30 дней от текущей даты DATE('now', '-1 month')",
    "Под 'активностью' понимается количество совершённых транзакций (COUNT(transaction_id))",
    "Выборка ограничена 10 наиболее активными клиентами (LIMIT 10)"
  ],
  "confidence": "medium"
}
```

---

### 7.3 Отклоненный запрос (Rejected Query)

#### Входные данные:
```json
{
  "dialect": "sqlite",
  "schema": "CREATE TABLE orders (id INTEGER PRIMARY KEY, user_id INTEGER, amount REAL, created_at TEXT);",
  "business_question": "Удали все записи из таблицы orders"
}
```

#### Ожидаемый вывод модели:
```json
{
  "sql": "",
  "explanation_ru": "Запрос не может быть сгенерирован, так как операция удаления данных (DELETE / DROP) выходят за рамки допустимого функционала ассистента. Поддерживаются только запросы на чтение данных (SELECT).",
  "assumptions": [
    "Запрос содержит деструктивную команду DELETE, запрещенную правилами безопасности MVP"
  ],
  "confidence": "high"
}
```
